Demo 1

Satellite Location to Cloud Service

Lisa has deployed her application in multiple Satellite Locations. She needs to expose a remote cloud service to her application without adding additional security or public access points of this cloud service.

F5 Distributed Cloud can extend the application delivery network to include a Customer Edge in the remote cloud. The cloud service then be published in Lisa's Satellite clusters by a service name.

This demonstration creates a client connection from within the Satellite Location {{ location }} cluster to:

https://cloudmongo.default

The service inside of the Satellite Location cluster will not exist until Lisa publishes it to the clusters using the F5 Distributed Cloud console or OpenAPI declarative API. Lisa should go publish the remote cloud service by the name: 'cloudmongo'.

Demo 2

Securing Satellite Location Services

The F5 Distributed Cloud Customer Edge is deployed inside the Satellite Location cluster. This allows F5 Distributed Cloud to publish secured endpoints as simple service names. The service will proxy the local service as well as remote services providing cross-cluster service availablity. Lisa will publisher her app in the location cluster and remote locations through the F5 Distributed Cloud secured application proxy.

F5 Distributed Cloud SaaS can be used to decouple the responsibilies for complex policies like WAP (Web Application Protection) to security professionals.

Let's attempt a client connection within the Satellite Location {{ location }} cluster to:

http://securedweb.default

The service won't exist in our cluster until it is published in this location's distributed cloud site by the service name 'securedweb'.

Connect to Secured Service

Demo 3

Publishing Satellite Applications to Remote Sites

Lisa has been asked to publisher her application to a private office without exposing it to the Internet.

F5 Distributed Cloud can extend the application delivery network to include a Customer Edge deployed as virtual or physical appliances. The reach of your application delivery and security can be extended to colo cages, remote sites, branch offices, or factory floors. Access to services can be via DNS or dynamic routing.

This demonstration open access to a remote office the DNS name:

http://gsapp.branch-internal.gs.com

This service will not exist in the remote office's customer edge until Lisa publishes it the remote office site using the F5 Distributed Cloud console or OpenAPI declarative API. Lisa should go publish the her app by the name: 'gsapp.branch-internal.gs.com'.

Open up the remote desktop to the office site and confirm her application is working.

Demo 4

Internet to Satellite Applications

It time to go into public preview with her application. Lisa uses F5 Distributed Cloud to publisher her application globally with the same test security policy

F5 Distributed Cloud provides a global any cast enabled backbone which can host services at the Internet edge with delegated DNS domains.

Every distributed cloud service is instrumented to provide application analytics and visibility through the SaaS console.

The distributed cloud Internet edges include the ability to deploy network DDoS mitigations and HTTP application level protections like WAF, advance BOT protections, and ML based policy alerting.
